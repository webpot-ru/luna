# Project Workflow

Этот документ является source of truth по рабочему процессу в проекте LunaCards.

## Базовый порядок работы

Для любой нетривиальной задачи порядок такой:

1. Изучить структуру проекта.
2. Открыть [docs/README.md](README.md).
3. Найти профильный source of truth документ.
4. Проверить существующие решения, ограничения и возможные противоречия.
5. Если изменение затрагивает зафиксированные правила или архитектуру, сначала предложить изменение пользователю и получить разрешение.
6. Внести изменения.
7. Обновить все затронутые документы в том же цикле.
8. Проверить, что документация и реализация не расходятся.

## Предложения по изменению

Если появляется идея изменить уже принятое решение, нельзя молча менять документацию или код. Нужно:

1. Кратко описать текущее правило.
2. Объяснить, зачем его менять.
3. Описать последствия изменения.
4. Получить явное подтверждение пользователя.
5. После подтверждения обновить все затронутые документы и реализацию.

## Когда документацию нужно обновлять обязательно

Документацию нужно обновлять вместе с изменениями в:

- модели данных карточек;
- списке языков;
- правилах перевода;
- правилах артиклей, рода и транскрипции;
- структуре примеров;
- spreadsheet-выдаче, экспорте, Google Sheets, CSV, JSON или seed-данных;
- правилах создания и проверки карточек.

## Правило против повторного переделывания

Перед началом работы по карточкам нужно проверить [Decision Log](decision-log.md). Если решение уже принято, его нельзя переобсуждать или менять в реализации без явного подтверждения пользователя.

Если появляется причина изменить принятое решение, нужно:

1. указать старое решение;
2. объяснить, почему оно больше не подходит;
3. предложить новое решение;
4. получить подтверждение пользователя;
5. обновить [Decision Log](decision-log.md) и все затронутые source of truth документы.

## Безопасное удаление

Файлы и папки нельзя удалять напрямую. Если объект нужно убрать из активной структуры проекта, его нужно переместить в `Trash/<timestamp>/<original-relative-path>` через:

```bash
scripts/move-to-trash.sh <relative-path>
```

Скрипт принимает только относительный путь внутри проекта и отказывается работать с абсолютными или небезопасными путями.

## Оптимизация токенов и фоновые задачи (Token-Saving Background Tasks)

Для экономии лимитов запросов (RPM/TPM) и контекстных токенов API Gemini при выполнении длительных скриптов (например, генерация видео, пакетный экспорт или тяжелый QA-аудит) агент должен использовать следующий рабочий процесс:

1. **Фоновый запуск**: Использовать `run_command` с коротким временем ожидания (например, `WaitMsBeforeAsync: 500-1000` мс), чтобы запустить процесс в фоне.
2. **Освобождение хода (Yield)**: Сразу прекратить вызов инструментов и завершить свой ход. Не запускать циклы опроса (`manage_task status`) и не слать пользователю промежуточных сообщений.
3. **Реактивный подъем (Wakeup)**: Система сама разбудит агента новым сообщением, когда фоновый процесс завершит выполнение.
4. **Итоговый отчет**: Получив уведомление о завершении, агент делает финальную сверку результатов и выдает пользователю готовый отчет.
